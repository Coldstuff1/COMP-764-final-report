\section{Introduction}
% ===========================================================

High-Level Synthesis (HLS) has become an essential methodology in
modern digital hardware design, enabling the automatic translation of
high-level algorithmic specifications written in C/C++ into synthesizable
hardware accelerators. A foundational phase of any HLS pipeline is
\emph{scheduling}---the process of mapping algorithmic operations to
specific clock cycles---and the choice of scheduling paradigm has
far-reaching consequences for latency, throughput, silicon area, and
power consumption.

\emph{Static scheduling} (SS) resolves every operation-to-cycle
assignment at compile time, producing a lean, centralised Finite State
Machine--Datapath (FSMD) with minimal control overhead~\cite{9378784}.
This enables aggressive resource sharing and instruction-level
parallelism on regular workloads, but forces pessimistic assumptions
under data-dependent branches or variable-latency memory accesses.
\emph{Dynamic scheduling} (DS), as implemented in
Dynamatic~\cite{10.1145/3373087.3375391}, generates elastic circuits
where every inter-component arc carries \textit{valid}/\textit{ready}
handshaking signals, allowing operations to fire as soon as their inputs
are available. This runtime adaptability benefits irregular kernels but
incurs per-arc overhead and forgoes compile-time resource sharing.
As Cheng et al.\ demonstrate in the DASS framework, neither paradigm
dominates universally~\cite{9378784}.

Rigorous comparison has historically been confounded by toolchain
heterogeneity. The recent maturation of MLIR~\cite{lattner2021mlir}
and CIRCT~\cite{circt} changes this: Kim et al.~\cite{10.1145/3689790}
show that static and dynamic accelerator IRs share substantial
structural commonality and can coexist within a single substrate,
enabling a toolchain-neutral, controlled comparison. This work seizes
that opportunity by grounding both the HIR~\cite{10.1145/3623278.3624767}
static pipeline and the Dynamatic dynamic pipeline in a shared
MLIR/CIRCT build environment and evaluating them on a common benchmark
suite, thereby isolating the scheduling paradigm as the variable under
study.

Achieving this required overcoming non-trivial engineering challenges.
The contributions of this work are as follows.

\begin{itemize}
  \item \textbf{Unified MLIR/CIRCT build environment.} A four-way
        codebase integration reconciles the divergent dependency trees
        of Dynamatic~v2.0 and an incomplete HIR-based pipeline,
        enabling both tools to build within a single CMake environment.

  \item \textbf{Pragma system extensions to Polygeist for HIR.}
        The Polygeist C-to-MLIR frontend is extended with a custom
        \texttt{\#pragma HLS} handler that parses \texttt{PIPELINE},
        \texttt{INTERFACE}, \texttt{BIND\_STORAGE}, and
        \texttt{EXTERN\_FUNC} directives as first-class MLIR
        attributes, eliminating manual post-hoc IR editing.

  \item \textbf{\texttt{AffineToHIRPrep} pass.} A new MLIR pass
        bridges the \texttt{hls.*} attribute vocabulary emitted by the
        pragma handler and the HIR-native attributes expected by the
        \texttt{AffineToHIR} dialect conversion, enabling fully
        automated C-to-HIR compilation.

  \item \textbf{Pragma-annotated Dynamatic benchmark suite.} The
        Dynamatic C benchmark kernels are annotated with
        \texttt{\#pragma HLS} directives, producing the shared
        benchmark suite on which both pipelines are evaluated.
\end{itemize}

\paragraph{Motivating Example.}
Consider the FIR reduction kernel below, which accumulates pairwise
products through a unit-latency multiplier and a combinational adder
with a target Initiation Interval (II) of one.

\begin{figure}[h]
\begin{lstlisting}
#define N 1024
#pragma HLS extern_func variable=add_i32 latency=0
#pragma HLS extern_func variable=mul_i32 latency=1
int add_i32(int a, int b);
int mul_i32(int a, int b);

void fir(int di[N], int idx[N]) {
#pragma HLS INTERFACE port=di  storage_type=ram_1p rd_latency=1
#pragma HLS INTERFACE port=idx storage_type=ram_1p rd_latency=1
  int tmp = 0;
  for (int i = 0; i < N; i++) {
#pragma HLS pipeline II = 1
    tmp = add_i32(tmp, mul_i32(idx[i], di[N - 1 - i]));
  }
}
\end{lstlisting}
\caption{Motivating example: an affine reduction with a loop-carried
         accumulator and a target II of one.}
\label{lst:motiv}
\end{figure}

HIR's ILP-based modulo scheduler chains the load, multiply, and
accumulate into a fixed pipeline meeting the requested II, emitting an
FSMD with a single multiplier, a single adder, and no per-arc
handshaking~\cite{10.1145/3623278.3624767}. Dynamatic compiles the same
source into an elastic circuit where the loop-back path through
\texttt{tmp} routes tokens through fork, join, and buffer
components~\cite{10.1145/3373087.3375391}---infrastructure that is
the price of generality but yields no compensating gain on this regular
kernel, inflating area and lengthening the recurrence's critical
path~\cite{9378784}. This example motivates three empirical questions:
(1)~do HIR's static schedules match or exceed Dynamatic's cycle counts
on affine kernels? (2)~how large is Dynamatic's area premium in ALUTs,
registers, and I/O pins? (3)~does valid/ready arbitration reduce
achievable clock frequency relative to HIR's FSM-based controller?

\section{Related Work}
% ===========================================================

\textbf{Static scheduling via HIR.}
HIR~\cite{10.1145/3623278.3624767} embeds explicit time variables
directly into an MLIR-based IR, enabling synchronisation-free loop
pipelining without a hand-written FSM, but assumes fixed-latency
operations and predictable control flow. This work uses HIR as the
static evaluation pipeline, extending its frontend with a pragma handler
to target the Dynamatic benchmark suite.

\textbf{Dynamic scheduling via Dynamatic.}
Dynamatic~\cite{10.1145/3373087.3375391} translates C/C++ into elastic
circuits communicating exclusively through valid/ready handshaking,
tolerating variable-latency events at the cost of per-arc overhead,
reduced resource sharing, and potential critical-path lengthening. This
work uses Dynamatic as the dynamic evaluation pipeline.

\textbf{Combining both paradigms: DASS.}
DASS~\cite{9378784} identifies regions amenable to static scheduling and
wraps them as black boxes inside an otherwise dynamic dataflow graph,
recovering ${\approx}74\%$ of the area savings of a fully static design
while achieving $135\%$ of a fully dynamic design's performance.
However, DASS operates across two separate tool flows (Vivado HLS and
Dynamatic) without a shared IR, making it difficult to attribute
observed differences solely to the scheduling strategy. This work is
motivated by DASS's insight but grounds both schedulers in a unified
MLIR/CIRCT substrate for a controlled, toolchain-neutral comparison.

\textbf{Unified compiler infrastructure.}
MLIR~\cite{lattner2021mlir} and CIRCT~\cite{circt} provide a shared
substrate where static and dynamic scheduling dialects coexist.
Kim et al.~\cite{10.1145/3689790} show that static and dynamic
accelerator IRs share substantial structural commonality, motivating the
comparative evaluation in this paper.

\section{Methodology}
\label{sec:methodology}
% ===========================================================

\subsection{Codebase Integration}

A prerequisite for a controlled comparison is a common toolchain
foundation. Dynamatic~v2.0 and the HIR static scheduling
pipeline~\cite{10.1145/3623278.3624767} were built on incompatible
snapshots of the MLIR/CIRCT ecosystem~\cite{circt,lattner2021mlir},
each maintaining its own fork of CIRCT and Polygeist with bespoke
dialect extensions and pinned dependency versions. A systematic
four-way integration was required: the Dynamatic forks of CIRCT and
Polygeist were adopted as authoritative baselines, and all HIR-specific
dialect definitions, scheduling passes, and lowering infrastructure were
ported forward. This involved updating CMake configurations and LLVM API
call sites, ensuring non-conflicting dialect registration between the
\texttt{HIR} and \texttt{Handshake} dialects, and reconciling
differences in the affine IR shapes emitted by the two Polygeist forks.
Following this integration, both pipelines build and execute within a
single CMake environment.

\subsection{HLS Pragma Annotations for the HIR Pipeline}
\label{sec:pragma-handler}

The HIR frontend extends the Polygeist C-to-MLIR ingestion pass with a
custom \texttt{\#pragma HLS} handler recognising six directives divided
into \emph{region pragmas} (attached to loop or function bodies) and
\emph{variable pragmas} (keyed to a named identifier). Memory binding
pragmas (\texttt{BIND\_STORAGE}) instruct the buffer allocation pass to
target BRAM or LUT-RAM primitives and forward read/write latency values
to the ILP scheduler as \texttt{result\_delays} annotations on every
\texttt{affine.load} and \texttt{affine.store}. Region pragmas
(\texttt{PIPELINE II=\textit{n}}) attach the attribute
\texttt{hls.PIPELINE\_II} to the corresponding \texttt{affine.for}
operation, which the scheduler uses to constrain the
\texttt{hir.next\_iter} offset. Interface and extern-function pragmas
model external memory port contracts and IP pipeline depths,
respectively.

\subsection{The \texttt{AffineToHIRPrep} Pass}
\label{sec:affine-to-hir-prep}

The \texttt{AffineToHIRPrep} pass runs immediately before the main
\texttt{AffineToHIR} dialect conversion, bridging the \texttt{hls.*}
attribute vocabulary emitted by the pragma handler and the HIR-native
vocabulary expected downstream. The pass operates in four sequential
steps: (1)~external function declarations have \texttt{hls.INTERFACE\_LATENCY}
renamed to \texttt{hir.delay} and are hoisted to the top of the module;
(2)~\texttt{memref} arguments and local \texttt{alloca} operations have
flat \texttt{hls.*} latency attributes repackaged into
\texttt{hir.memref.ports} arrays; (3)~every \texttt{affine.load},
\texttt{affine.store}, and \texttt{func.call} is annotated with a
\texttt{hir.memref\_port} index and a \texttt{result\_delays} array
derived from the port descriptors established in step~2; and
(4)~\texttt{hls.PIPELINE\_II} attributes on \texttt{affine.for} loops
are renamed to the \texttt{II} token consumed by the ILP scheduler.
After all four steps, every \texttt{hls.*} attribute has been consumed
and the IR is structurally ready for \texttt{AffineToHIR} without
further manual editing. Figure~\ref{fig:prep-before-after} shows a
side-by-side view of the \texttt{gesummv} kernel IR immediately before
and after the pass; each transformation step described above can be
located by comparing the corresponding lines across the two listings.

\section{Evaluation}
% ===========================================================

\subsection{Benchmark Selection and Experimental Setup}

All benchmark kernels are drawn from the Dynamatic test suite---a
collection of small-to-medium C kernels designed to exercise elastic
dataflow scheduling~\cite{10.1145/3373087.3375391}. HIR's own
benchmarks were excluded because their structural complexity produces
Dynamatic netlists that exceed the placement-and-routing capacity of
the available FPGA target, rendering a fair comparison impossible.
The Dynamatic source files were annotated with \texttt{\#pragma HLS}
directives (\texttt{PIPELINE}, \texttt{INTERFACE}, \texttt{BIND\_STORAGE})
as the sole modification; algorithmic content and array dimensions
remain unchanged. Both pipelines were synthesised with Quartus Prime
targeting the Arria~10AX115N4F45I3SG FPGA at a 500~MHz clock
constraint; correctness was verified in ModelSim.

\subsection{Results and Analysis}
\label{sec:results}

Figure~\ref{fig:all-results} presents post-synthesis ratios
(Dynamatic$/$HIR) across the seven benchmarks; ratios above one
indicate Dynamatic consumes more of that resource.

\paragraph{Area: ALUTs and Registers.}
The area overhead of dynamic scheduling is substantial and consistent
(Figures~\ref{fig:aluts}~and~\ref{fig:registers}). ALUT ratios range
from ${\approx}2\times$ (\texttt{simple\_example}) to $6.5\times$
(\texttt{sumi3\_mem}), with a mean of $4.75\times$. Register ratios are
more pronounced: \texttt{FIR} and \texttt{sumi3\_mem} exceed $11\times$
and $12.5\times$, respectively. These costs arise because every
inter-component arc in Dynamatic carries valid/ready handshaking logic,
and the absence of compile-time resource sharing forces functional unit
replication that a static scheduler would avoid~\cite{9378784}.

\paragraph{I/O: GPIO Pin Usage.}
Dynamatic designs require $3.5$--$5.4\times$ the GPIO pins of their HIR
counterparts (Figure~\ref{fig:pins}). The ratio is uniform across
benchmarks, reflecting a structural property: Dynamatic's top-level
entity exposes valid, ready, and data ports for every function argument
and memory interface, whereas HIR's FSM multiplexes memory accesses
through a compact internal port map.

\paragraph{Frequency: $F_{\max}$.}
$F_{\max}$ ratios cluster near unity (Figure~\ref{fig:fmax}). Four
benchmarks match or slightly exceed HIR's frequency; the remaining
three (\texttt{FIR} at ${\approx}0.74\times$, \texttt{sumi3\_mem} at
${\approx}0.79\times$, \texttt{matvec} at ${\approx}0.87\times$) show
modest degradation attributable to the combinational depth of
handshaking arbitration on feedback-heavy paths. The $F_{\max}$ penalty
is at most 26\% and is largely absorbed by Quartus Prime's register
retiming.

\paragraph{Latency: Number of Cycles.}
In six of seven benchmarks the cycle-count ratio exceeds $1.0$
(Figure~\ref{fig:cycles}). \texttt{FIR} and \texttt{sumi3\_mem} are
worst at ${\approx}2.95\times$; \texttt{stencil\_2d}, \texttt{matrix},
and \texttt{matvec} incur moderate penalties of $1.85\times$,
$1.5\times$, and $1.4\times$. \texttt{vector\_rescale} achieves cycle
parity (${\approx}1.0\times$)---a single regular loop nest where dynamic
scheduling adds overhead without compensating flexibility. The advantage
of HIR stems from its ability to compute tight, fixed IIs via SDC-based
modulo scheduling, whereas Dynamatic's runtime token coordination
manifests as extra cycles on statically predictable workloads.

\paragraph{Combined Interpretation.}
Taken together, the five metrics paint a coherent picture. On
regular, affine-dominated kernels, the statically scheduled HIR pipeline
dominates across area and latency while the two approaches achieve
comparable maximum frequencies. The area savings are dramatic---often
exceeding $4\times$ in ALUTs and $5\times$ in registers---because HIR's
compile-time resource sharing and FSM-based control eliminate the per-arc
handshaking infrastructure inherent to elastic circuits, while its
SDC-based modulo scheduler achieves tighter initiation intervals than
Dynamatic's runtime token flow~\cite{9378784}. These results confirm the
predictions of Cheng et al.: the critical insight is not the directional
outcome---static scheduling wins on regular workloads---but the
\emph{magnitude} of the gap, which underscores the practical cost of the
dynamic paradigm's generality when that generality is not exploited by
the workload.

\section{Discussion}
% ===========================================================

On regular, affine-dominated kernels, the statically scheduled HIR
pipeline dominates across area and execution latency while both
paradigms achieve comparable maximum frequencies. These findings
recapitulate, within a unified IR substrate, the qualitative trends
reported by Cheng et al.~\cite{9378784} and Josipović et
al.~\cite{10.1145/3373087.3375391}, while quantifying the
\emph{magnitude} of the gap: area and latency penalties are an order
of magnitude larger than sometimes assumed, and persist even under a
shared compiler infrastructure. The key mechanism is straightforward:
HIR's compile-time resource sharing and FSM-based control eliminate the
per-arc handshaking infrastructure inherent to elastic circuits, and its
SDC-based modulo scheduler achieves tighter IIs than Dynamatic's runtime
token flow on workloads without data-dependent control divergence.

The interpretation of these results must, however, be tempered by the
conditions under which they were obtained. The benchmark suite was drawn
from kernels designed to exercise Dynamatic's scheduling capabilities,
yet the kernels selected for evaluation are precisely those whose control
flow is regular and memory access patterns are statically
analysable---exactly the workload regime in which static scheduling is
predicted to dominate~\cite{9378784}. The dynamic paradigm's principal
claimed advantage---runtime adaptation to data-dependent control
divergence and variable-latency memory accesses---is not exercised by
any benchmark in the present suite, and consequently its overheads
appear without compensating benefit. The principal contribution of this
comparison is therefore to quantify the \emph{magnitude} of that
overhead under controlled, toolchain-neutral conditions, demonstrating
that it persists even when both pipelines share a common compiler
infrastructure.

\subsection{Limitations}
\label{sec:limitations}

The results must be qualified by several methodological constraints.
The evaluation is limited to seven affine, regular kernels---precisely
the regime where static scheduling is predicted to dominate~\cite{9378784}.
Dynamic scheduling's principal advantage---runtime adaptation to
data-dependent control divergence---is not exercised by any benchmark
in the suite; its overheads therefore appear without compensating
benefit. Additionally, Dynamatic was originally tuned for Xilinx FPGAs
and the Vivado toolchain, whereas both pipelines were synthesised here
through Quartus Prime on an Intel Arria~10 device, introducing
backend-specific optimisation behaviour that may penalise Dynamatic
relative to its native target. All metrics were extracted from a single
post-synthesis report per benchmark under a single pragma-prescribed
configuration, with no design-space exploration. Finally, since
Dynamatic operates on unannotated source while HIR requires
user-supplied pragmas, there is no mechanical guarantee that both
compilers are presented with semantically equivalent scheduling intent.

\subsection{Future Work}
\label{sec:future-work}

The most immediate extension is a benchmark suite designed to stress
dynamic scheduling's strengths: kernels with data-dependent loop bounds,
non-affine memory accesses, sparse linear algebra, and graph traversals.
Such workloads are expected to narrow or invert the cycle-count
advantage observed for HIR, and quantifying this crossover within a
unified IR substrate would provide a more complete characterisation of
the paradigm trade-off~\cite{9378784}. Complementary to this, the
shared MLIR/CIRCT substrate and Kim et al.'s IR-unification
results~\cite{10.1145/3689790} make a hybrid scheduler---integrating
HIR-style statically scheduled regions as black-box components within a
Dynamatic-style elastic dataflow graph---an attainable engineering goal,
eliminating the cross-tool boundary that DASS currently traverses.
Coupling such a hybrid scheduler with an automatic analysis of
loop-carried dependence patterns, predicate-driven control flow, and
memory access regularity in the MLIR affine IR would close the loop on
a fully automated, pragma-free, paradigm-aware HLS flow.

Further extensions include vendor-neutral synthesis across Quartus Prime
and Vivado, design-space exploration sweeping pipelining factors and
clock targets to yield Pareto frontiers in the area--throughput plane,
and power characterisation incorporating Quartus PowerPlay or Vivado XPE
to assess the energy cost of per-arc handshaking under realistic
activity profiles.

\section{Conclusion}
% ===========================================================

This work has presented a controlled, toolchain-neutral comparison
between static and dynamic HLS scheduling, grounded in a unified
MLIR/CIRCT substrate hosting both the HIR static
pipeline~\cite{10.1145/3623278.3624767} and the Dynamatic dynamic
pipeline~\cite{10.1145/3373087.3375391}. The required engineering
contributions---a four-way dependency reconciliation, an HLS pragma
handler for Polygeist, an \texttt{AffineToHIRPrep} bridge pass, and a
pragma-annotated Dynamatic benchmark suite---collectively enable
C-to-hardware compilation through both flows from a single codebase.

Across seven kernels synthesised under identical 500~MHz constraints,
the statically scheduled HIR pipeline produced consistently smaller
and faster hardware: mean ALUTs and registers were reduced by
${\approx}4.75\times$ and up to $12.5\times$; GPIO pins fell by
$3.5$--$5.4\times$; and execution latency fell by $1.4$--$2.95\times$
on six of the seven kernels. Maximum frequencies were broadly
comparable, with Dynamatic incurring at most a 26\% penalty.

These results strongly suggest that committing an entire design to
dynamic scheduling is wasteful when the program contains substantial
regular sub-regions, while committing entirely to static scheduling
forfeits throughput gains on irregular sub-regions~\cite{9378784}.
The unified MLIR/CIRCT foundation laid here makes a hybrid
paradigm-aware HLS scheduler---one that gets most of the area benefit
of static scheduling and most of the performance benefit of dynamic
scheduling within a single automated flow---an attainable next step.
