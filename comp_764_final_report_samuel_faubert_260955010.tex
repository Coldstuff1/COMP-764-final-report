\section{Introduction}

\subsection{Problem Statement}

High-Level Synthesis (HLS) has become an essential methodology in modern digital hardware design, abstracting away the intricacies of Register Transfer Level (RTL) development by enabling the automatic translation of high-level algorithmic specifications (e.g., C/C++) into synthesizable circuits. The primary objective of any HLS compiler is to generate high-performance hardware accelerators optimized for strict constraints on latency, throughput, silicon area (LUTs, FFs, DSPs), and power consumption. 

A foundational and highly consequential phase in the HLS pipeline is scheduling—the process of mapping algorithmic operations to specific clock cycles. Historically, HLS frameworks have favored static scheduling \cite{9378784} %
. By constructing a centralized Finite State Machine (FSM) at compile time, static schedulers can execute deterministic, highly parallelized datapaths with a minimal control footprint. However, this methodology struggles with irregular workloads; when a circuit encounters variable memory latencies or data-dependent control flow, static scheduling relies on pessimistic assumptions that degrade overall performance \cite{9378784} %
. Conversely, dynamic scheduling paradigms generate dataflow-based elastic circuits where decentralized handshaking protocols dictate execution at runtime, gracefully tolerating unpredictability but often incurring localized area and routing overheads \cite{10.1145/3373087.3375391} %
.

With the recent maturation of the Multi-Level Intermediate Representation (MLIR) and the Circuit IR (CIRCT) ecosystems, hardware compiler infrastructure has evolved to support sophisticated, specialized abstractions for diverse scheduling methodologies \cite{10.1145/3689790} %
. This convergence presents a unique opportunity to evaluate disparate scheduling techniques using equitable compiler backend technologies. 

The primary goal of this paper is to empirically investigate the benefits and hardware trade-offs between static and dynamic scheduling by executing controlled experiments across two distinct MLIR/CIRCT-based generation pipelines. Specifically, I compare the static scheduling abstractions provided by the Hardware Intermediate Representation (HIR) \cite{10.1145/3623278.3624767} %
 against the dynamically scheduled, synchronous dataflow circuits generated by the Dynamatic framework \cite{10.1145/3373087.3375391} %
. By compiling a shared set of benchmarks through both pipelines and analyzing the resulting synthesized hardware, I aim to systematically quantify the architectural, performance, and resource trade-offs inherent to each scheduling paradigm.

\section{Related Work}

\subsection{MLIR and CIRCT: A Modern Compiler Infrastructure for HLS}

The Multi-Level Intermediate Representation (MLIR) is a compiler
infrastructure developed to address fragmentation in domain-specific
and general-purpose language compilation~\cite{lattner2021mlir}.
Rather than imposing a single level of abstraction, MLIR defines a
unified framework in which multiple \emph{dialects}---each a namespaced
collection of operations, types, and attributes---can coexist and
interoperate within the same compilation unit, enabling
\emph{progressive lowering} from high-level, semantically rich
representations down to target-specific hardware
descriptions~\cite{lattner2021mlir}.
Building on this foundation, the Circuit IR Compilers and Tools (CIRCT)
project adapts MLIR's methodology for hardware design and HLS,
introducing a hierarchy of dialects spanning from control-flow-centric
abstractions (\texttt{SCF}, \texttt{Affine}, \texttt{Handshake}) down
to structural, synthesizable constructs (\texttt{HW}, \texttt{Comb},
\texttt{SV})~\cite{circt}.

This infrastructure is significant to the HLS scheduling debate because
it cleanly separates scheduling strategy from code generation.
Static scheduling approaches such as HIR~\cite{10.1145/3623278.3624767}
operate as MLIR dialects where every operation carries an explicit,
compiler-assigned time variable, exposing schedule verification,
bitwidth reduction, and delay coalescing as composable pass-based
transformations.
Dynamic scheduling likewise benefits: the \texttt{Handshake} dialect
provides first-class IR support for valid/ready handshaking, enabling
Dynamatic to represent and optimise elastic circuits entirely within the
MLIR framework~\cite{10.1145/3373087.3375391}.
As demonstrated by Kim et al.~\cite{10.1145/3689790}, static and dynamic
intermediate languages share substantial structural commonality and can
be unified under a single compiler framework, motivating the comparative
evaluation in this paper: by grounding both pipelines in the same
MLIR/CIRCT ecosystem, observed differences in latency, area, and
throughput can be attributed to the scheduling strategy rather than
to backend divergence.

\subsection{Static Scheduling in High-Level Synthesis}

In High-Level Synthesis (HLS), scheduling is the fundamental process of
allocating operations from a high-level algorithmic description to
specific clock cycles~\cite{9378784}.
Traditional HLS tools predominantly rely on static scheduling, where
the mapping of each operation to a clock cycle is determined entirely
at compile time~\cite{9378784}.
This paradigm targets a Finite State Machine with Datapath (FSMD)
architecture, where the synthesised FSM orchestrates multiplexers and
registers cycle-by-cycle according to the predetermined schedule.
Static scheduling excels in applications with regular control flow and
predictable memory access patterns, enabling aggressive optimisations
such as resource sharing and loop pipelining with a deterministic
Initiation Interval~(II).
However, it requires pessimistic assumptions under unpredictable memory
dependencies or irregular control flow, which has motivated the
emergence of dynamic scheduling~\cite{10.1145/3373087.3375391}.

\subsubsection{Static Scheduling via the HIR Intermediate Representation}

A prominent example of formalising static scheduling within the IR is
the Hardware Intermediate Representation (HIR), an MLIR-based mid-level
IR tailored for hardware accelerator design of affine
workloads~\cite{10.1145/3623278.3624767}.
HIR departs from the conventional \emph{datapath\,+\,FSM} model by
adopting an explicit \emph{datapath\,+\,schedule} representation.
Every execution region---a loop body or function definition---is
assigned a \emph{time variable}, and all operations within that scope
are scheduled at a constant offset relative to the region's dynamic
start time~\cite{10.1145/3623278.3624767}.
This mechanism maps different dynamic execution instances of the same
static operation to distinct clock cycles without requiring the frontend
to manually construct FSMs.

By treating the schedule as a first-class language construct, HIR
supports synchronisation-free, fine-grained parallelism spanning
instruction-level parallelism, loop pipelining, and overlapped task
execution~\cite{10.1145/3623278.3624767}.
Embedding scheduling contracts into function signatures further allows
the compiler to statically verify data dependencies and automatically
synthesise the required FSMs during lowering to SystemVerilog.

\subsection{Dynamic Scheduling and Elastic Circuits}

While static scheduling achieves high efficiency in regular
applications, its reliance on compile-time assumptions limits its
applicability to programs with unpredictable latencies, such as
irregular memory accesses or data-dependent control
flow~\cite{9378784}.
Dynamic scheduling abandons the centralised, cycle-by-cycle global
controller in favour of a distributed execution model where operations
are triggered strictly by data availability at
runtime~\cite{10.1145/3373087.3375391}.

\subsubsection{Dynamatic and Dataflow-Based Execution}

A prominent realisation of dynamic scheduling in HLS is the Dynamatic
framework, an open-source compiler that translates C/C++ into
dynamically scheduled synchronous dataflow
circuits~\cite{10.1145/3373087.3375391}.
Rather than an explicit schedule, Dynamatic constructs \textit{Elastic
Circuits}: hardware components communicate exclusively via local
\textit{valid}/\textit{ready} handshaking, exchanging data tokens
precisely when all conditions for a transaction are
satisfied~\cite{10.1145/3373087.3375391}.
Control flow is realised without a centralised FSM through four
specialised routing primitives: \textsc{Fork} (token fan-out),
\textsc{Join} (operand synchronisation), \textsc{Branch} (conditional
steering), and \textsc{Merge} (non-deterministic arbitration).

By interconnecting these primitives, Dynamatic constructs a dataflow
graph that resolves control dependencies dynamically, extracting
fine-grained instruction-level parallelism and tolerating variable
latency events---such as cache misses or unpredictable loop
bounds---that would stall a statically scheduled
pipeline~\cite{10.1145/3373087.3375391}.
The cost is twofold: every dataflow arc introduces handshaking logic
that increases area and may lengthen the critical path, and the absence
of compile-time scheduling precludes functional unit
sharing~\cite{9378784,10.1145/3373087.3375391}.

\section{Overview and Motivation}
% ===========================================================

Two competing imperatives govern the design of any HLS scheduler:
minimal hardware area---best served by aggressive compile-time resource
sharing---and maximal throughput---best served by runtime adaptability.
Static scheduling (SS) resolves this tension conservatively by fixing
every operation-to-cycle mapping at compile time, yielding a lean FSMD
with no runtime control overhead but forcing pessimistic assumptions
under irregular workloads~\cite{9378784}.
Dynamic scheduling (DS) defers all decisions to runtime via local
\textit{valid}/\textit{ready} handshaking, producing circuits that adapt
on-the-fly at the cost of per-arc overhead and lost resource-sharing
opportunities~\cite{10.1145/3373087.3375391}.
As Cheng~et~al.\ demonstrate in the DASS framework, neither paradigm
dominates universally~\cite{9378784}.

% ----------------------------------------------------------
\subsection{Motivating Example}
% ----------------------------------------------------------

Consider the kernel in Listing~\ref{lst:motiv}, where array \texttt{b}
is conditionally updated depending on a runtime predicate derived from
\texttt{a[i]}.

\begin{figure}[h]
\begin{lstlisting}
#define N 128
void conditional_acc(int a[N], int b[N]) {
  for (int i = 0; i < N; i++) {
    if (a[i] > 0)
      b[i] = b[i] + a[i];  // data-dependent write
  }
}
\end{lstlisting}
\caption{Motivating example: a loop with a data-dependent conditional write.}
\label{lst:motiv}
\end{figure}

\paragraph{Static schedule.}
Because the compiler cannot determine which iterations will execute the
write, the static scheduler conservatively sets the loop II to the
full load-modify-store latency, stalling the pipeline on every iteration
where \texttt{a[i]\,$\leq$\,0} and the write is skipped.

\paragraph{Dynamic schedule.}
The Dynamatic elastic circuit imposes no global stall: iterations that
skip the write bypass it immediately, and backpressure activates only
on genuine read-after-write hazards.
Throughput adapts to the actual data distribution at the cost of larger
circuit area from per-arc handshake registers and routing
primitives~\cite{10.1145/3373087.3375391}.

% ----------------------------------------------------------
\subsection{Research Questions}
% ----------------------------------------------------------

This example motivates three empirical questions addressed in this study.
(1)~\textbf{Throughput:} For regular, affine kernels, does HIR's static
schedule match or exceed Dynamatic's cycle counts, and does DS yield
measurable gains for irregular kernels?
(2)~\textbf{Area:} How large is the area premium---in ALUTs, Registers,
and DSPs---incurred by Dynamatic's per-arc handshaking relative to HIR's
statically scheduled datapath?
(3)~\textbf{Frequency:} Does the combinational overhead of
\textit{valid}/\textit{ready} arbitration in Dynamatic lengthen the
critical path and reduce the achievable clock frequency relative to
HIR's FSM-based controller?
Both pipelines are compiled under identical synthesis constraints
targeting the same FPGA device, with all metrics extracted from
post-synthesis reports; the shared MLIR/CIRCT substrate minimises
toolchain divergence as a confounding variable.



\section{Methodology}

\subsection{Codebase Integration: Unifying Two Independent HLS Pipelines}

A prerequisite for a controlled comparison between static and dynamic
scheduling is that both pipelines share a common toolchain foundation,
ensuring that observed differences in generated hardware quality are
attributable to the scheduling paradigm rather than to backend divergence.
Achieving this required reconciling two independent research projects—
Dynamatic~v2.0 (pre-release v0.3)~\cite{10.1145/3373087.3375391} and an
incomplete HIR-based static scheduling pipeline~\cite{10.1145/3623278.3624767}—
built on incompatible snapshots of the MLIR/CIRCT
ecosystem~\cite{circt,lattner2021mlir}.

Each project maintains its own fork of both the CIRCT compiler
infrastructure and the Polygeist C-to-MLIR frontend, with bespoke dialect
extensions, pass modifications, and pinned dependency versions that could
not simultaneously coexist. Resolving these incompatibilities requires a
systematic four-way codebase integration: the Dynamatic forks of CIRCT and
Polygeist are adopted as the authoritative baselines, and all HIR-specific
dialect definitions, scheduling passes, and lowering infrastructure are
ported forward onto those baselines. This involves updating CMake
configurations and LLVM API call sites to match Dynamatic's pinned versions,
ensuring non-conflicting dialect registration between the \texttt{HIR} and
\texttt{Handshake} dialects, and reconciling differences in the affine IR
shapes emitted by the two Polygeist forks.

Following this integration, both pipelines build and execute within a single
CMake environment, providing the unified compiler on which I conduct all
subsequent experiments in this study.

\subsection{HLS Pragma Annotations for the HIR Pipeline}

The HIR frontend extends the Polygeist C-to-MLIR ingestion pass with a
custom \texttt{\#pragma HLS} handler that recognises six directives,
divided into \emph{region pragmas} (attached to loop or function bodies)
and \emph{variable pragmas} (keyed to a named identifier via a
\texttt{variable=} or \texttt{port=} clause).

\paragraph{Region pragmas.}
\texttt{\#pragma HLS PIPELINE II=\textit{n}} attaches the attribute
\texttt{hls.PIPELINE\_II} to the enclosing \texttt{affine.for} operation;
during the \texttt{AffineToHIRPrep} pass this is renamed to the HIR
attribute \texttt{II} and consumed by the ILP-based scheduler to
constrain the \texttt{hir.next\_iter} offset, enforcing the
user-specified Initiation Interval directly in the IR.

\paragraph{Variable pragmas.}
\texttt{\#pragma HLS INTERFACE port=\textit{p} storage\_type=\{ram\_1p
$|$ ram\_2p\} rd\_latency=\textit{r} wr\_latency=\textit{w}} specifies
the memory-interface contract for an external buffer argument; the
\texttt{AffineToHIRPrep} pass translates it into a
\texttt{hir.memref.ports} array attribute that drives the synthesis of
address, data, and enable buses in the generated SystemVerilog.
\texttt{\#pragma HLS BIND\_STORAGE variable=\textit{v}
impl=\{bram$|$lutram\} rd\_latency=\textit{r} wr\_latency=\textit{w}}
applies the same latency semantics to a locally allocated
\texttt{memref.alloca}, additionally specifying the target FPGA memory
primitive via the \texttt{mem\_kind} attribute consumed by the buffer
allocation pass.
Finally, \texttt{\#pragma HLS EXTERN\_FUNC variable=\textit{f}
latency=\textit{l}} annotates an externally provided Verilog IP (e.g.\
a floating-point unit) with its pipeline depth; the value is lowered to
a \texttt{hir.delay} attribute via the \texttt{result\_delays} annotation,
allowing the ILP scheduler to insert the correct number of pipeline stages
between the call site and any consuming operation.
% ----------------------------------------------------------
\subsection{The \texttt{AffineToHIRPrep} Pass}
\label{sec:affine-to-hir-prep}
% ----------------------------------------------------------

A third contribution of this work is the \texttt{AffineToHIRPrep}
MLIR pass. This pass runs immediately before the main \texttt{AffineToHIR}
dialect conversion and serves as a normalisation and annotation
layer between the \texttt{hls.*} attribute vocabulary emitted by the
pragma handler and the HIR-native attribute vocabulary expected by
the downstream conversion.

The pass operates in three sequential stages.
First, it normalises all \emph{external} function declarations by
renaming \texttt{hls.INTERFACE\_LATENCY} to \texttt{hir.delay} on
arguments and results, synthesising default argument names where
absent, and hoisting the declarations to the top of the module so
that every call site can resolve its callee during conversion.
Second, it processes each non-external function by translating the
\texttt{hls.INTERFACE\_*} and \texttt{hls.BIND\_STORAGE\_*}
attributes placed by the pragma handler into HIR's
\texttt{hir.memref.ports} port-descriptor arrays and
\texttt{mem\_kind} implementation attributes, renaming
\texttt{hls.PIPELINE\_II} to the \texttt{II} attribute directly
consumed by the ILP scheduler, and annotating every
\texttt{affine.load} and \texttt{affine.store} with
\texttt{result\_delays} and \texttt{hir.memref\_port} indices
derived from the port descriptors.
It additionally promotes rank-0 \texttt{memref} values---arising from
scalar C variables that Polygeist lowers to zero-dimensional
buffers---to \texttt{memref<1xT>} with register-style port
annotations, as HIR has no representation for zero-dimensional
memories.
Finally, all intermediate \texttt{hls.*} attributes are stripped,
leaving only well-formed HIR attributes for the subsequent
conversion pass.
Together, these transformations eliminate the need for manual
post-hoc IR editing that the original HIR compiler flow
requires~\cite{10.1145/3623278.3624767}, and ensure that the
\texttt{AffineToHIR} conversion receives a structurally valid
and fully annotated module.

\section{Evaluation}

\subsection{Benchmark Selection and Pragma Adaptation}

To enable a controlled comparison between the two compilation
pipelines, a common benchmark suite is required. The benchmark
kernels used in this study are drawn exclusively from the
Dynamatic test suite, a collection of small-to-medium C kernels
specifically designed to exercise the scheduling capabilities of
elastic dataflow circuits~\cite{10.1145/3373087.3375391}.

Although HIR defines its own set of benchmarks---complex,
multi-loop affine workloads including systolic matrix
multiplication, multi-stage image-processing pipelines, and
producer-consumer loop nests~\cite{10.1145/3623278.3624767}---these
kernels are not compatible with the Dynamatic compilation flow.
Their structural complexity and memory access patterns produce
elastic circuits whose resource footprint exceeds the capacity of
the FPGA devices available through Intel Quartus Prime, the
target synthesis platform used in this evaluation. Attempting to
synthesise these designs results in circuits that cannot be placed
and routed on the available target devices, rendering a fair
comparison impossible. Consequently, HIR's own benchmarks are
excluded from the evaluation, and all experiments are conducted
on the Dynamatic benchmark suite.

Because the Dynamatic benchmarks are plain C programs annotated
only for the Dynamatic frontend, they require adaptation before
they can be ingested by the HIR compilation pipeline. This is
accomplished by adding \texttt{\#pragma HLS} directives to each
kernel, leveraging the pragma handler developed as part of this
work. Specifically, \texttt{PIPELINE}
and \texttt{UNROLL} pragmas are inserted on loop nests to specify
the desired pipelining and unrolling factors, \texttt{INTERFACE}
pragmas are added to function arguments to declare memory port
types, and \texttt{BIND\_STORAGE} pragmas are applied to local
arrays to assign explicit port configurations. These annotations
are the sole modifications made to the original Dynamatic source
files; the algorithmic content and array dimensions remain
unchanged, preserving the integrity of the comparison.
