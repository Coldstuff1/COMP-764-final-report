\section{Introduction}
% ===========================================================

High-Level Synthesis (HLS) has become an essential methodology in modern
digital hardware design, enabling the automatic translation of high-level
algorithmic specifications written in C/C++ into synthesizable hardware
accelerators. A foundational and consequential phase of any HLS pipeline
is \emph{scheduling}---the process of mapping algorithmic operations to
specific clock cycles---and the choice of scheduling paradigm has
far-reaching consequences for latency, throughput, silicon area, and
power consumption.

Historically, HLS frameworks have favoured \emph{static scheduling} (SS),
which resolves every operation-to-cycle assignment at compile time and
produces a lean, centralised Finite State Machine--Datapath (FSMD) with
minimal control overhead~\cite{9378784}.
The strength of this approach lies in its ability to aggressively share
hardware resources and exploit instruction-level parallelism when the
workload is regular and predictable.
However, when a circuit encounters data-dependent branches or
variable-latency memory accesses, a static scheduler is forced to adopt
pessimistic assumptions---e.g., stalling the entire pipeline on every
iteration regardless of whether the stall-inducing path is actually
taken---thereby sacrificing performance on irregular workloads.

\emph{Dynamic scheduling} (DS) addresses this limitation by deferring
control decisions to runtime.
Frameworks such as Dynamatic~\cite{10.1145/3373087.3375391} generate
synchronous dataflow elastic circuits in which every inter-component arc
carries \textit{valid}/\textit{ready} handshaking signals, allowing
operations to fire as soon as their inputs are available without waiting
for a centralised controller.
This runtime adaptability enables the circuit to exploit actual data
distributions and bypass idle execution paths, yielding throughput gains
on irregular kernels.
The cost, however, is non-trivial: per-arc handshaking registers and
routing primitives increase area and can lengthen the critical path,
while the absence of compile-time resource allocation limits the
opportunities for hardware sharing.
As Cheng et al.\ demonstrate through the DASS hybrid framework, neither
paradigm dominates universally---fully static designs are more area
efficient, while fully dynamic designs outperform them on
control-intensive code~\cite{9378784}.

Comparing these two paradigms rigorously has historically been
confounded by toolchain heterogeneity: static schedulers and dynamic
compilers have typically relied on incompatible intermediate
representations and backend flows, making it difficult to attribute
observed differences to the scheduling strategy rather than to incidental
implementation choices.
The recent maturation of the Multi-Level Intermediate Representation
(MLIR) and the Circuit IR (CIRCT) ecosystems changes this picture.
Kim et al.\ have shown that static and dynamic accelerator IRs share
substantial structural commonality and can coexist within a single
MLIR/CIRCT substrate~\cite{10.1145/3689790}, creating the conditions
for a toolchain-neutral, controlled comparison.
This work seizes that opportunity: by grounding both the
Hardware Intermediate Representation (HIR)~\cite{10.1145/3623278.3624767}
static pipeline and the Dynamatic dynamic pipeline in the same MLIR/CIRCT
build environment, compiling a shared benchmark suite through both, and
analysing the resulting synthesized hardware, we isolate the effect of
the scheduling paradigm on latency, throughput, and FPGA resource usage.

Enabling this comparison required overcoming several non-trivial
engineering challenges: the two toolchains have divergent dependency
trees, incompatible dialect vocabularies, and different expectations about
how source-level annotations are expressed.
The contributions of this work are as follows.

\begin{itemize}

  \item \textbf{Unified MLIR/CIRCT build environment.}
        We perform a four-way codebase integration that reconciles the
        divergent dependency trees of Dynamatic~v2.0 and an incomplete
        HIR-based pipeline, enabling both tools to build and execute
        within a single, coherent CMake environment.

  \item \textbf{Pragma system extensions to Polygeist for HIR generation.}
        We extend the Polygeist C-to-MLIR frontend with a custom
        \texttt{\#pragma HLS} handler that parses
        \texttt{PIPELINE}, \texttt{INTERFACE}, \texttt{BIND\_STORAGE},
        and \texttt{EXTERN\_FUNC} directives and attaches them as
        first-class MLIR attributes, eliminating the need for manual
        post-hoc IR editing required by the original HIR flow.

  \item \textbf{\texttt{AffineToHIRPrep} pass.}
        We develop a new MLIR pass that bridges the \texttt{hls.*}
        attribute vocabulary emitted by the pragma handler and the
        HIR-native attributes expected by the \texttt{AffineToHIR}
        dialect conversion, collectively enabling fully automated
        C-to-HIR compilation.

  \item \textbf{Pragma-annotated Dynamatic benchmark suite for HIR.}
        We annotate the Dynamatic C benchmark kernels with
        \texttt{\#pragma HLS} directives, producing the shared benchmark
        suite on which both pipelines are evaluated and all
        area/throughput comparisons are grounded.

\end{itemize}

\section{Related Work}
% ===========================================================

\textbf{Static scheduling via HIR.}
HIR~\cite{10.1145/3623278.3624767} is an MLIR-based intermediate
representation for hardware accelerator design that formalises static
scheduling by embedding explicit time variables directly into the IR,
enabling synchronisation-free loop pipelining and overlapped task
execution without a hand-written FSM.
While highly area-efficient for affine, regular workloads, HIR assumes
fixed-latency operations and predictable control flow, making it
ill-suited to kernels with data-dependent branches or variable-latency
memory accesses.
The present work uses HIR as one of the two evaluation pipelines,
extending its compilation flow with a pragma-based frontend to target
the Dynamatic benchmark suite.

\textbf{Dynamic scheduling via Dynamatic.}
Dynamatic~\cite{10.1145/3373087.3375391} is an open-source HLS compiler
that translates C/C++ into dynamically scheduled elastic circuits, where
components communicate exclusively through valid/ready handshaking and
require no centralised controller.
This design tolerates variable-latency events and irregular control flow
that would stall a static pipeline, but incurs per-arc handshaking
overhead that increases area, reduces resource sharing, and can lengthen
the critical path.
This work uses Dynamatic as the second evaluation pipeline, and
quantifying these area and frequency costs relative to a statically
scheduled baseline is a primary empirical objective.

\textbf{Combining both paradigms: DASS.}
DASS~\cite{9378784} has pioneered the combination of static and dynamic
scheduling within a single HLS circuit, identifying regions amenable
to static scheduling via user-supplied pragmas and wrapping them as
black-box components inside an otherwise dynamically scheduled dataflow
graph.
Evaluated across realistic benchmarks, DASS recovers approximately 74\%
of the area savings of a fully static design while achieving 135\% of
the performance of a fully dynamic one---demonstrating that neither
paradigm dominates universally.
However, DASS operates across two separate tool flows (Vivado HLS for
static regions, Dynamatic for dynamic regions) without a shared IR
substrate, making it difficult to attribute observed differences solely
to the scheduling strategy.
The present work is motivated by DASS's insight but instead grounds both
schedulers within the unified MLIR/CIRCT ecosystem, enabling a
controlled, toolchain-neutral comparison.

\textbf{Unified compiler infrastructure.}
MLIR~\cite{lattner2021mlir} and the CIRCT project~\cite{circt} provide a
shared compiler infrastructure for hardware design in which static and
dynamic scheduling dialects coexist.
Kim et al.~\cite{10.1145/3689790} have shown that static and dynamic
accelerator IRs share substantial structural commonality and can be
unified under a single framework, motivating the comparative evaluation
in this paper: by grounding both pipelines in the same MLIR/CIRCT
substrate, observed differences in latency, area, and throughput can be
attributed to the scheduling paradigm rather than to backend divergence.

\section{Overview and Motivation}
% ===========================================================

Two competing imperatives govern the design of any HLS scheduler:
minimal hardware area---best served by aggressive compile-time resource
sharing---and maximal throughput---best served by runtime adaptability.
Static scheduling (SS) resolves this tension conservatively by fixing
every operation-to-cycle mapping at compile time, yielding a lean FSMD
with no runtime control overhead but forcing pessimistic assumptions
under irregular workloads~\cite{9378784}.
Dynamic scheduling (DS) defers all decisions to runtime via local
\textit{valid}/\textit{ready} handshaking, producing circuits that adapt
on-the-fly at the cost of per-arc overhead and lost resource-sharing
opportunities~\cite{10.1145/3373087.3375391}.
As Cheng~et~al.\ demonstrate in the DASS framework, neither paradigm
dominates universally~\cite{9378784}.

% ----------------------------------------------------------
\subsection{Motivating Example}
% ----------------------------------------------------------

Consider the kernel in Listing~\ref{lst:motiv}, where array \texttt{b}
is conditionally updated depending on a runtime predicate derived from
\texttt{a[i]}.

\begin{figure}[h]
\begin{lstlisting}
#define N 128
void conditional_acc(int a[N], int b[N]) {
  for (int i = 0; i < N; i++) {
    if (a[i] > 0)
      b[i] = b[i] + a[i];  // data-dependent write
  }
}
\end{lstlisting}
\caption{Motivating example: a loop with a data-dependent conditional write.}
\label{lst:motiv}
\end{figure}

\paragraph{Static schedule.}
Because the compiler cannot determine which iterations will execute the
write, the static scheduler conservatively sets the loop II to the
full load-modify-store latency, stalling the pipeline on every iteration
where \texttt{a[i]\,$\leq$\,0} and the write is skipped.

\paragraph{Dynamic schedule.}
The Dynamatic elastic circuit imposes no global stall: iterations that
skip the write bypass it immediately, and backpressure activates only
on genuine read-after-write hazards.
Throughput adapts to the actual data distribution at the cost of larger
circuit area from per-arc handshake registers and routing
primitives~\cite{10.1145/3373087.3375391}.

% ----------------------------------------------------------
\subsection{Research Questions}
% ----------------------------------------------------------

This example motivates three empirical questions addressed in this study.
(1)~\textbf{Throughput:} For regular, affine kernels, does HIR's static
schedule match or exceed Dynamatic's cycle counts, and does DS yield
measurable gains for irregular kernels?
(2)~\textbf{Area:} How large is the area premium---in ALUTs, Registers,
and DSPs---incurred by Dynamatic's per-arc handshaking relative to HIR's
statically scheduled datapath?
(3)~\textbf{Frequency:} Does the combinational overhead of
\textit{valid}/\textit{ready} arbitration in Dynamatic lengthen the
critical path and reduce the achievable clock frequency relative to
HIR's FSM-based controller?
Both pipelines are compiled under identical synthesis constraints
targeting the same FPGA device, with all metrics extracted from
post-synthesis reports; the shared MLIR/CIRCT substrate minimises
toolchain divergence as a confounding variable.



\section{Methodology}

\subsection{Codebase Integration: Unifying Two Independent HLS Pipelines}

A prerequisite for a controlled comparison between static and dynamic
scheduling is that both pipelines share a common toolchain foundation,
ensuring that observed differences in generated hardware quality are
attributable to the scheduling paradigm rather than to backend divergence.
Achieving this required reconciling two independent research projects—
Dynamatic~v2.0 (pre-release v0.3)~\cite{10.1145/3373087.3375391} and an
incomplete HIR-based static scheduling pipeline~\cite{10.1145/3623278.3624767}—
built on incompatible snapshots of the MLIR/CIRCT
ecosystem~\cite{circt,lattner2021mlir}.

Each project maintains its own fork of both the CIRCT compiler
infrastructure and the Polygeist C-to-MLIR frontend, with bespoke dialect
extensions, pass modifications, and pinned dependency versions that could
not simultaneously coexist. Resolving these incompatibilities requires a
systematic four-way codebase integration: the Dynamatic forks of CIRCT and
Polygeist are adopted as the authoritative baselines, and all HIR-specific
dialect definitions, scheduling passes, and lowering infrastructure are
ported forward onto those baselines. This involves updating CMake
configurations and LLVM API call sites to match Dynamatic's pinned versions,
ensuring non-conflicting dialect registration between the \texttt{HIR} and
\texttt{Handshake} dialects, and reconciling differences in the affine IR
shapes emitted by the two Polygeist forks.

Following this integration, both pipelines build and execute within a single
CMake environment, providing the unified compiler on which all
subsequent experiments in this study are conducted.

\subsection{HLS Pragma Annotations for the HIR Pipeline}

The HIR frontend extends the Polygeist C-to-MLIR ingestion pass with a
custom \texttt{\#pragma HLS} handler that recognises six directives,
divided into \emph{region pragmas} (attached to loop or function bodies)
and \emph{variable pragmas} (keyed to a named identifier via a
\texttt{variable=} or \texttt{port=} clause).
To make the role of each directive concrete, consider the annotated
kernel shown in Listing~\ref{lst:pragma-example}, which is drawn from
the \texttt{funcIx} benchmark used in the evaluation.

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{figs/pragmas.png}
  \caption{Pragma-annotated C source for the \texttt{funcIx} kernel.
           Each \texttt{\#pragma HLS} directive targets either a loop
           region or a named variable and is consumed by the
           \texttt{AffineToHIRPrep} pass to produce well-formed HIR.}
  \label{lst:pragma-example}
\end{figure}

\noindent
Walking through the listing from top to bottom illustrates how the
pragma categories interact:

\paragraph{Memory binding pragma (line~27).}
The outermost annotation,
\begin{quote}
\texttt{\#pragma HLS bind\_storage variable = w impl = bram type = ram\_2p
rd\_latency = 1 wr\_latency = 1}
\end{quote}
targets the local weight array \texttt{w}.  The \texttt{impl=bram} clause
instructs the buffer allocation pass to map this array onto a block-RAM
primitive rather than a distributed LUT-RAM, while \texttt{type=ram\_2p}
selects a true-dual-port interface.  The latency attributes
\texttt{rd\_latency=1} and \texttt{wr\_latency=1} are forwarded to the
ILP scheduler as \texttt{result\_delays} annotations on every
\texttt{affine.load} and \texttt{affine.store} that references \texttt{w},
so that the schedule correctly accounts for the one-cycle read and write
latency of the BRAM primitive.

\paragraph{Region (pipeline) pragmas (lines~31, 33, 36, 38).}
Each of the four nested \texttt{affine.for} loops carries a
\texttt{PIPELINE II=\textit{n}} directive:
\begin{itemize}
  \item \texttt{II=1300} on the outermost row loop (line~31) sets a loose
        initiation interval, effectively imposing no pipelining pressure at
        that level and allowing the inner loops to dominate scheduling.
  \item \texttt{II=40} on the column loop (line~33) constrains how
        frequently a new column iteration may be initiated.
  \item \texttt{II=18} and \texttt{II=6} on the reduction loops over
        \texttt{u} and \texttt{v} (lines~36 and~38, respectively) tighten
        the pipeline aggressively on the innermost accumulation, where
        throughput is most critical.
\end{itemize}
Each directive attaches the attribute \texttt{hls.PIPELINE\_II} to the
corresponding \texttt{affine.for} operation; during the
\texttt{AffineToHIRPrep} pass this attribute is renamed to \texttt{II}
and consumed by the ILP-based scheduler to constrain the
\texttt{hir.next\_iter} offset, enforcing the user-specified Initiation
Interval directly in the IR.

\paragraph{Scop delimiters (lines~25 and~45).}
The \texttt{\#pragma scop} and \texttt{\#pragma endscop} markers bracket
the affine region of interest.  These are standard Polygeist directives,
not HLS-specific, and signal to the frontend which loop nest should be
lowered into the \texttt{Affine} dialect rather than remaining as
unstructured \texttt{scf.for} operations.  They are included here because
they are a prerequisite for the pragma handler: the HLS directives are
only meaningful inside an affine region where loop bounds are
statically analysable.

\paragraph{Variable pragmas.}
\texttt{\#pragma HLS INTERFACE port=\textit{p} storage\_type=\{ram\_1p
$|$ ram\_2p\} rd\_latency=\textit{r} wr\_latency=\textit{w}} specifies
the memory-interface contract for an external buffer argument; the
\texttt{AffineToHIRPrep} pass translates it into a
\texttt{hir.memref.ports} array attribute that drives the synthesis of
address, data, and enable buses in the generated SystemVerilog.
\texttt{\#pragma HLS BIND\_STORAGE variable=\textit{v}
impl=\{bram$|$lutram\} rd\_latency=\textit{r} wr\_latency=\textit{w}}
applies the same latency semantics to a locally allocated
\texttt{memref.alloca}, additionally specifying the target FPGA memory
primitive via the \texttt{mem\_kind} attribute consumed by the buffer
allocation pass.
Finally, \texttt{\#pragma HLS EXTERN\_FUNC variable=\textit{f}
latency=\textit{l}} annotates an externally provided Verilog IP (e.g.\
a floating-point unit) with its pipeline depth; the value is lowered to
a \texttt{hir.delay} attribute via the \texttt{result\_delays} annotation,
allowing the ILP scheduler to insert the correct number of pipeline stages
between the call site and any consuming operation.

\subsection{The \texttt{AffineToHIRPrep} Pass}
\label{sec:affine-to-hir-prep}
% ----------------------------------------------------------

\begin{figure*}[htbp]
  \centering
  \includegraphics[width=\linewidth]{figs/affine.png}
  \subcaption{Before \texttt{AffineToHIRPrep} (Listing~A).}
  \label{fig:prep-before}

  \vspace{1em}

  \includegraphics[width=\linewidth]{figs/prepped.png}
  \subcaption{After \texttt{AffineToHIRPrep} (Listing~B).}
  \label{fig:prep-after}

  \caption{\texttt{gesummv} kernel IR before and after the
           \texttt{AffineToHIRPrep} pass. Each transformation step
           described in the text can be located by comparing
           the corresponding lines across the two listings.}
  \label{fig:prep-before-after}
\end{figure*}

The \texttt{AffineToHIRPrep} pass runs immediately before the main
\texttt{AffineToHIR} dialect conversion and serves as a normalisation
and annotation bridge between the \texttt{hls.*} attribute vocabulary
emitted by the pragma handler (Section~\ref{sec:pragma-handler}) and the
HIR-native attribute vocabulary expected by the downstream conversion.
Without this pass, the \texttt{AffineToHIR} conversion would encounter
unknown attributes and fail to produce valid HIR.

To make the transformation concrete, Figure~\ref{fig:prep-before-after}
shows a side-by-side view of the \texttt{gesummv} kernel IR immediately
\emph{before} (Listing~A, left) and \emph{after} (Listing~B, right) the
pass is applied. The pass operates in four sequential steps, each of
which can be traced between the two listings.

\paragraph{Step~1 --- External function normalisation.}
Every externally defined function declaration (e.g.\ \texttt{@mul\_i32}
and \texttt{@add\_i32}, lines~1--2 of Listing~A) carries
\texttt{hls.INTERFACE\_LATENCY} attributes on its arguments and result.
The pass renames each of these to \texttt{hir.delay}
(lines~1--2 of Listing~B), which is the attribute the HIR scheduler uses
to model the pipeline depth of external IP blocks.
In addition, a \texttt{resultNames} attribute (\texttt{["out"]}) is
synthesised where absent, satisfying an assertion in the
\texttt{AffineToHIR} conversion that every external callee must expose
named results. All normalised declarations are then hoisted to the top of
the module so that call sites encountered later in the IR walk can always
resolve their callee.

\paragraph{Step~2 --- Memory argument and local buffer translation.}
Function arguments of \texttt{memref} type (e.g.\ \texttt{\%arg2} and
\texttt{\%arg3}, lines~5--6 of Listing~A) carry the flat
\texttt{hls.INTERFACE\_PORT}, \texttt{hls.INTERFACE\_STORAGE\_TYPE}, and
\texttt{hls.INTERFACE\_LATENCY} attributes attached by the pragma handler.
The pass replaces these with a \texttt{hir.memref.ports} array whose
dictionaries distinguish read and write ports by key
(\texttt{rd\_latency}/\texttt{wr\_latency}), as required by the HIR
memory-port model (lines~5--6 of Listing~B).
The same transformation applies to local \texttt{memref.alloca}
operations (line~8 of Listing~A): the \texttt{hls.BIND\_STORAGE\_*}
latency values are repackaged into a \texttt{hir.memref.ports} array and
the implementation hint (\texttt{hls.BIND\_STORAGE\_IMPL = "reg"}) is
mapped to the \texttt{mem\_kind} attribute consumed by the buffer
allocation pass (line~8 of Listing~B).
Scalar arguments likewise have their \texttt{hls.INTERFACE\_LATENCY}
renamed to \texttt{hir.delay}.

\paragraph{Step~3 --- Memory access and call-site annotation.}
Every \texttt{affine.load}, \texttt{affine.store}, and
\texttt{func.call} must carry two annotations before
\texttt{AffineToHIR} can lower them: a \texttt{hir.memref\_port} index
identifying which port of the operand's \texttt{hir.memref.ports} array
to use, and a \texttt{result\_delays} array specifying the access latency
seen by consuming operations. The pass derives both values from the port
descriptors established in Step~2. For instance, the \texttt{affine.load}
operations in the inner loop body (lines~14--15 of Listing~A) carry only
a raw \texttt{hls.INTERFACE\_LATENCY} hint from the pragma handler; after
the pass (lines~14--15 of Listing~B) they carry
\texttt{hir.memref\_port = 0} (selecting the first read port) and
\texttt{result\_delays = [1]} (one-cycle read latency). The
\texttt{func.call} sites (e.g.\ lines~16--17 of Listing~B) receive a
\texttt{result\_delays} value derived from the \texttt{hir.delay}
attribute of the callee's result established in Step~1.

\paragraph{Step~4 --- Pipeline II renaming.}
The \texttt{hls.PIPELINE\_II} attribute placed by the pragma handler on
\texttt{affine.for} loops (visible at the closing braces of the loop
nests in Listing~A, e.g.\ \texttt{\{hls.PIPELINE\_II = 1 : i64\}} and
\texttt{\{hls.PIPELINE\_II = 18 : i64\}}) is renamed to the \texttt{II}
token (Listing~B, same locations) directly consumed by the ILP scheduler
to constrain the \texttt{hir.next\_iter} offset and enforce the
user-specified Initiation Interval.

After all four steps complete, every \texttt{hls.*} attribute has been
consumed and replaced by its HIR-native equivalent. The IR emitted by
the pass is structurally ready for the \texttt{AffineToHIR} dialect
conversion to proceed without further modification or manual IR editing.

\section{Evaluation}

\subsection{Benchmark Selection and Pragma Adaptation}

To enable a controlled comparison between the two compilation
pipelines, a common benchmark suite is required. The benchmark
kernels used in this study are drawn exclusively from the
Dynamatic test suite, a collection of small-to-medium C kernels
specifically designed to exercise the scheduling capabilities of
elastic dataflow circuits~\cite{10.1145/3373087.3375391}.

Although HIR defines its own set of benchmarks---complex,
multi-loop affine workloads including systolic matrix
multiplication, multi-stage image-processing pipelines, and
producer-consumer loop nests~\cite{10.1145/3623278.3624767}---these
kernels are not compatible with the Dynamatic compilation flow.
Their structural complexity and memory access patterns produce
elastic circuits whose resource footprint exceeds the capacity of
the FPGA devices available through Intel Quartus Prime, the
target synthesis platform used in this evaluation. Attempting to
synthesise these designs results in circuits that cannot be placed
and routed on the available target devices, rendering a fair
comparison impossible. Consequently, HIR's own benchmarks are
excluded from the evaluation, and all experiments are conducted
on the Dynamatic benchmark suite.

Because the Dynamatic benchmarks are plain C programs annotated
only for the Dynamatic frontend, they require adaptation before
they can be ingested by the HIR compilation pipeline. This is
accomplished by adding \texttt{\#pragma HLS} directives to each
kernel, leveraging the pragma handler developed as part of this
work. Specifically, \texttt{PIPELINE}
and \texttt{UNROLL} pragmas are inserted on loop nests to specify
the desired pipelining and unrolling factors, \texttt{INTERFACE}
pragmas are added to function arguments to declare memory port
types, and \texttt{BIND\_STORAGE} pragmas are applied to local
arrays to assign explicit port configurations. These annotations
are the sole modifications made to the original Dynamatic source
files; the algorithmic content and array dimensions remain
unchanged, preserving the integrity of the comparison.
