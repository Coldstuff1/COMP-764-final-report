\section{Introduction}

\subsection{Problem Statement}

High-Level Synthesis (HLS) has become an essential methodology in modern digital hardware design, abstracting away the intricacies of Register Transfer Level (RTL) development by enabling the automatic translation of high-level algorithmic specifications (e.g., C/C++) into synthesizable circuits. The primary objective of any HLS compiler is to generate high-performance hardware accelerators optimized for strict constraints on latency, throughput, silicon area (LUTs, FFs, DSPs), and power consumption. 

A foundational and highly consequential phase in the HLS pipeline is scheduling—the process of mapping algorithmic operations to specific clock cycles. Historically, HLS frameworks have favored static scheduling \cite{9378784} %
. By constructing a centralized Finite State Machine (FSM) at compile time, static schedulers can execute deterministic, highly parallelized datapaths with a minimal control footprint. However, this methodology struggles with irregular workloads; when a circuit encounters variable memory latencies or data-dependent control flow, static scheduling relies on pessimistic assumptions that degrade overall performance \cite{9378784} %
. Conversely, dynamic scheduling paradigms generate dataflow-based elastic circuits where decentralized handshaking protocols dictate execution at runtime, gracefully tolerating unpredictability but often incurring localized area and routing overheads \cite{10.1145/3373087.3375391} %
.

With the recent maturation of the Multi-Level Intermediate Representation (MLIR) and the Circuit IR (CIRCT) ecosystems, hardware compiler infrastructure has evolved to support sophisticated, specialized abstractions for diverse scheduling methodologies \cite{10.1145/3689790} %
. This convergence presents a unique opportunity to evaluate disparate scheduling techniques using equitable compiler backend technologies. 

The primary goal of this paper is to empirically investigate the benefits and hardware trade-offs between static and dynamic scheduling by executing controlled experiments across two distinct MLIR/CIRCT-based generation pipelines. Specifically, I compare the static scheduling abstractions provided by the Hardware Intermediate Representation (HIR) \cite{10.1145/3623278.3624767} %
 against the dynamically scheduled, synchronous dataflow circuits generated by the Dynamatic framework \cite{10.1145/3373087.3375391} %
. By compiling a shared set of benchmarks through both pipelines and analyzing the resulting synthesized hardware, I aim to systematically quantify the architectural, performance, and resource trade-offs inherent to each scheduling paradigm.

\section{Related Work}
\subsection{MLIR and CIRCT: A Modern Compiler Infrastructure for HLS}

The Multi-Level Intermediate Representation (MLIR) is a compiler
infrastructure originally developed to address the fragmentation endemic
to domain-specific and general-purpose language compilation~\cite{lattner2021mlir}.
Rather than imposing a single level of abstraction, MLIR defines a unified
framework in which multiple \emph{dialects}---each a namespaced collection
of operations, types, and attributes---can coexist and interoperate within
the same compilation unit. This design enables \emph{progressive lowering},
where a program is incrementally transformed from high-level, semantically
rich representations down to target-specific hardware descriptions, with
each dialect capturing the information most relevant to the optimizations
performed at that stage~\cite{lattner2021mlir}.

Building directly on this foundation, the Circuit IR Compilers and Tools
(CIRCT) project is an LLVM sub-project and umbrella initiative that adapts
MLIR's methodology specifically for hardware design and high-level
synthesis~\cite{circt}. CIRCT introduces a hierarchy of hardware-oriented
dialects spanning different levels of abstraction: from high-level,
control-flow-centric dialects such as \texttt{SCF} and \texttt{Affine},
through the \texttt{Handshake} dialect that encodes latency-insensitive,
token-based dataflow semantics, down to structural dialects such as
\texttt{HW}, \texttt{Comb}, and \texttt{SV} that correspond closely to
synthesizable SystemVerilog constructs~\cite{10.1145/3623278.3624767}.
The availability of the \texttt{export-verilog} pass within CIRCT provides
a common, reusable backend for emitting synthesizable RTL from any
dialect that lowers into the \texttt{HW}/\texttt{SV} layer.

The significance of this infrastructure for the HLS scheduling debate lies
in its ability to cleanly separate the concerns of scheduling strategy from
code generation. Static scheduling approaches, such as those embodied by the
Hardware Intermediate Representation (HIR)~\cite{10.1145/3623278.3624767},
operate as MLIR dialects in which every operation carries an explicit,
compiler-assigned time variable. The HIR compiler, built atop MLIR and
targeting CIRCT's structural dialects, demonstrates that an LLVM-like,
pass-based architecture can perform schedule verification, bitwidth
reduction, and delay coalescing as composable, reusable transformation
passes~\cite{10.1145/3623278.3624767}. Dynamic scheduling approaches
likewise benefit from the CIRCT ecosystem: the \texttt{Handshake} dialect
provides first-class IR support for the valid/ready handshaking protocol
underpinning dataflow circuits, enabling tools such as the modern
rewrite of Dynamatic to represent, analyse, and optimise elastic circuits
entirely within the MLIR framework~\cite{10.1145/3373087.3375391}.

A particularly relevant contribution of the CIRCT ecosystem to this work
is the unification it offers across scheduling paradigms. As demonstrated
by Kim et al.~\cite{10.1145/3689790}, static and dynamic intermediate
languages for accelerator generation share substantial structural
commonality and can be unified under a single compiler framework without
sacrificing the semantic distinctions that make each paradigm appropriate
for different workloads. This perspective motivates the comparative
evaluation presented in this paper: by grounding both the statically
scheduled HIR pipeline and the dynamically scheduled Dynamatic pipeline
in the same MLIR/CIRCT ecosystem, I am able to attribute observed
differences in latency, area, and throughput to the scheduling strategy
itself, rather than to disparities in backend code quality or target
toolchain.

\subsection{Static Scheduling in High-Level Synthesis}

In High-Level Synthesis (HLS), scheduling is the fundamental process of allocating operations from a high-level algorithmic description to specific clock cycles \cite{9378784} %
. Traditional HLS tools predominantly rely on static scheduling, where the mapping of each operation to a clock cycle is determined entirely at compile time \cite{9378784} %
. This paradigm typically targets a Finite State Machine with Datapath (FSMD) architecture, where the synthesized FSM acts as a controller that orchestrates multiplexers and registers cycle-by-cycle based on the predetermined schedule. Static scheduling excels in applications with regular control flow and predictable memory access patterns, enabling aggressive hardware optimizations such as resource sharing and highly optimized loop pipelining—often measured by a deterministic Initiation Interval (II). However, this methodology requires pessimistic assumptions when dealing with unpredictable memory dependencies or highly irregular control flow, which has largely motivated the emergence of dynamic scheduling architectures \cite{10.1145/3373087.3375391} %
.

\subsubsection{Static Scheduling via the HIR Intermediate Representation}

Recent advancements in HLS compiler infrastructure seek to formalize static scheduling directly within the intermediate representation (IR) to decouple the scheduling process from downstream logic synthesis. A prominent example is the Hardware Intermediate Representation (HIR), an MLIR-based mid-level IR explicitly tailored for hardware accelerator design of affine workloads \cite{10.1145/3623278.3624767} %
.

HIR fundamentally departs from the conventional ``datapath + FSM'' representation by modeling the hardware explicitly as ``datapath + schedule'' \cite{10.1145/3623278.3624767} %
. It achieves this abstraction by introducing the concept of \textit{time variables} \cite{10.1145/3623278.3624767} %
. Every execution region—such as a loop body or a function definition—is assigned a start time variable, and all operations within that scope are scheduled at a constant delay relative to this dynamic start time \cite{10.1145/3623278.3624767} %
. This mechanism allows the IR to map different dynamic execution instances of the same static operation to distinct clock cycles without requiring the frontend compiler to manually construct complex FSMs \cite{10.1145/3623278.3624767} %
.

By treating the schedule as a first-class language construct, HIR supports synchronization-free, fine-grained parallelism. It seamlessly represents high-level transformations like instruction-level parallelism, loop pipelining, and overlapped task execution without relying on runtime handshaking protocols \cite{10.1145/3623278.3624767} %
. Furthermore, embedding these precise scheduling contracts into function signatures allows the compiler to statically verify data dependencies and automatically synthesize the required FSMs during the final lowering to SystemVerilog \cite{10.1145/3623278.3624767} %
.

\subsection{Dynamic Scheduling and Elastic Circuits}

While static scheduling achieves high efficiency in regular applications, its reliance on compile-time assumptions limits its applicability to programs with unpredictable latencies, such as irregular memory accesses or data-dependent control flows \cite{9378784} %
. To overcome the pessimistic assumptions mandated by the traditional Finite State Machine with Datapath (FSMD) model, recent High-Level Synthesis (HLS) research has increasingly explored dynamic scheduling. Dynamic scheduling abandons the centralized, cycle-by-cycle global controller in favor of a distributed execution model where operations are triggered strictly by data availability at runtime \cite{10.1145/3373087.3375391} %
. 

\subsubsection{Dynamatic and Dataflow-Based Execution}

A prominent realization of dynamic scheduling in HLS is the Dynamatic framework, an open-source compiler that translates standard C/C++ into dynamically scheduled synchronous dataflow circuits \cite{10.1145/3373087.3375391} %
. Unlike the explicit ``datapath + schedule'' abstraction found in HIR, Dynamatic constructs \textit{Elastic Circuits}. In this paradigm, hardware components communicate exclusively via local handshaking protocols (e.g., \textit{valid} and \textit{ready} signals), exchanging data tokens precisely when all conditions for a transaction are satisfied \cite{10.1145/3373087.3375391} %
.

To faithfully execute complex, high-level control flow structures without a centralized FSM, Dynamatic employs a specialized set of dataflow routing components:
\begin{itemize}
    \item \textbf{Fork:} Duplicates a single input token to multiple downstream operations, ensuring synchronization by only firing once all receivers are ready.
    \item \textbf{Join:} Synchronizes multiple input tokens, ensuring an operation only fires when all required operands are present.
    \item \textbf{Branch:} Routes a data token to one of several output paths based on an evaluated boolean condition token, effectively steering control flow.
    \item \textbf{Merge:} Acts as a non-deterministic multiplexer, accepting a token from whichever input path becomes valid first.
\end{itemize}

By interconnecting these handshaking primitives, Dynamatic constructs a dataflow graph capable of resolving control dependencies dynamically. This elasticity inherently extracts fine-grained, instruction-level parallelism and allows the circuit to adapt its schedule on-the-fly. Consequently, dynamically scheduled circuits gracefully tolerate variable latency events—such as cache misses or unpredictable loop bounds—that would otherwise stall a statically scheduled pipeline \cite{10.1145/3373087.3375391} %
.



\section{Overview and Motivation}

\subsection{The Fundamental Scheduling Trade-Off}

Two competing imperatives govern the design of any High-Level Synthesis
(HLS) scheduler: the desire for minimal hardware area---best served by
aggressive compile-time resource sharing---and the desire for maximal
throughput---best served by runtime adaptability to unpredictable control
and memory behaviour.
Static scheduling (SS) resolves this tension conservatively by fixing the
allocation of every operation to a specific clock cycle during compilation,
producing a lean Finite State Machine with Datapath (FSMD) architecture
that requires no runtime control overhead~\cite{9378784}.
Dynamic scheduling (DS), by contrast, defers all scheduling decisions to
runtime: hardware components communicate exclusively through local
\textit{valid}/\textit{ready} handshaking signals and fire as soon as
their input tokens are available, yielding circuits capable of
adapting their effective schedule on-the-fly~\cite{10.1145/3373087.3375391}.
Neither paradigm is universally superior.
In this section, I develop a concrete motivating example that illustrates
precisely when and why the choice of scheduling strategy matters, and then
outline the experimental approach taken in this work to quantify those
trade-offs.

% ----------------------------------------------------------
\subsection{Static Scheduling via HIR}
% ----------------------------------------------------------

In statically scheduled HLS, the compiler constructs the complete
operation-to-cycle mapping prior to synthesis.
The Hardware Intermediate Representation (HIR)~\cite{10.1145/3623278.3624767}
embodies this paradigm within the MLIR ecosystem by replacing the
conventional \textit{datapath + FSM} abstraction with an explicit
\textit{datapath + schedule} representation.
Every execution region---a loop body, a function---is assigned a
\textit{time variable}, and every operation within that region is
scheduled at a statically known offset relative to its region's start
time~\cite{10.1145/3623278.3624767}.
The downstream compiler pass then mechanically synthesises the required
FSM from these scheduling contracts, without requiring the frontend to
manually construct control logic.

This approach excels in \textit{regular, affine workloads} where loop
bounds and memory access patterns are fully predictable at compile time.
Under such conditions, the scheduler can safely pipeline loops to their
minimum achievable Initiation Interval (II), share functional units across
non-overlapping operations, and produce circuits with a minimal area
footprint.
However, when a kernel contains data-dependent control flow or
variable-latency memory accesses, the static scheduler must adopt
\textit{pessimistic assumptions}---reserving extra pipeline stages or
inflating the II to accommodate the worst-case execution path---thereby
leaving performance on the table even when the worst case is rarely
encountered~\cite{9378784}.

% ----------------------------------------------------------
\subsection{Dynamic Scheduling via Dynamatic}
% ----------------------------------------------------------

Dynamatic~\cite{10.1145/3373087.3375391} represents the alternative
extreme.
Rather than encoding a global schedule into the circuit, it constructs
\textit{Elastic Circuits}: a dataflow graph in which every arc carries
\textit{valid} and \textit{ready} signals.
A component fires precisely when all required input tokens are valid and
the downstream consumer signals readiness; backpressure propagates
upstream automatically if a consumer is not ready.
Specialised routing primitives---\textsc{Fork}, \textsc{Join},
\textsc{Branch}, and \textsc{Merge}---implement conditional and
concurrent control flow without any centralised FSM.

This architecture is inherently resilient to variability: a cache miss or
a short-circuit loop exit simply stalls the affected path locally rather
than forcing a global pipeline flush.
In applications with irregular memory access patterns or data-dependent
loop bounds, this elasticity translates directly into lower cycle counts.
The cost is twofold: (i) every dataflow arc introduces handshaking logic,
increasing the per-channel area and potentially lengthening the critical
path, and (ii) the absence of compile-time scheduling makes resource
sharing across operations difficult, so functional units are generally
not reused~\cite{9378784,10.1145/3373087.3375391}.

% ----------------------------------------------------------
\subsection{Motivating Example}
% ----------------------------------------------------------

To make the trade-off concrete, consider the loop kernel in
Listing~\ref{lst:motiv}.
The array \texttt{b} is conditionally updated on every iteration
depending on a runtime predicate derived from \texttt{a[i]}.
The number of loop iterations, \texttt{N}, is fixed, but the latency of
the conditional write---and therefore the data-dependency chain from one
iteration to the next---varies at runtime.

\begin{figure}[h]
\begin{lstlisting}
  #define N 128
  void conditional_acc(int a[N], int b[N]) {
    for (int i = 0; i < N; i++) {
        if (a[i] > 0)
            // data-dependent write
            b[i] = b[i] + a[i];
    }
}
\end{lstlisting}
\caption{Motivating example: a loop with a data-dependent
         conditional write.}
\label{lst:motiv}
\end{figure}

\paragraph{Static schedule}
Because the compiler cannot determine at compile time which iterations
will execute the write, the static scheduler must conservatively assume
that \emph{every} iteration incurs the full load-modify-store latency.
It therefore sets the loop II to accommodate this worst-case path,
stalling the pipeline for cycles in which \texttt{a[i] $\leq$ 0} and
the write is skipped.

\paragraph{Dynamic schedule}
The Dynamatic elastic circuit imposes no global stall.
Iterations for which \texttt{a[i] $\leq$ 0} bypass the write path
immediately; backpressure only activates when a genuine read-after-write
hazard is detected at runtime.
Consequently, throughput adapts to the actual data distribution: sparse
conditional writes yield near-II\,=\,1 throughput, whereas dense writes
degrade gracefully to the memory-bound limit.
The penalty is a larger circuit area due to the per-arc handshake
registers and routing primitives required to implement the elastic
dataflow graph~\cite{10.1145/3373087.3375391}.

As Cheng~et~al.\ demonstrate in the context of the DASS framework, the
relative benefit of dynamic over static scheduling is strongly
workload-dependent, and neither paradigm dominates across all
benchmarks~\cite{9378784}.

% ----------------------------------------------------------
\subsection{Research Questions and Experimental Approach}
% ----------------------------------------------------------

The motivating example raises three concrete research questions that
I address empirically in this study:

\begin{enumerate}

  \item \textbf{Throughput under regularity vs.\ irregularity.}
        For kernels with fully affine, predictable access patterns, does
        HIR's static schedule achieve comparable or superior cycle counts
        to Dynamatic's elastic circuits?
        Conversely, for kernels with data-dependent control flow or
        irregular memory accesses, does dynamic scheduling yield
        measurable throughput improvements?

  \item \textbf{Area overhead of elastic control.}
        How large is the area premium---measured in Adaptive Look-Up Tables (ALUTs),
        Registers, and GPIO pins---incurred by Dynamatic's
        per-arc handshaking infrastructure relative to HIR's statically
        scheduled datapath?

  \item \textbf{Impact on achievable clock frequency.}
        Does the additional combinational logic introduced by
        \textit{valid}/\textit{ready} arbitration in Dynamatic lengthen
        the critical path and reduce the achievable clock frequency
        compared to HIR's FSM-based controller?

\end{enumerate}

To answer these questions, a shared set of benchmark kernels spanning
both regular and irregular workloads is compiled through both the HIR
and Dynamatic pipelines under identical synthesis constraints.
Both pipelines target the same FPGA device, and all hardware metrics
are extracted from post-synthesis reports, ensuring a controlled and
equitable comparison.
The MLIR/CIRCT ecosystem~\cite{10.1145/3689790,circt} provides a common
compiler substrate (after my intervention), reducing tool-chain 
divergence as a confounding variable.
The methodology, benchmarks, and synthesis environment are described in
detail in the following section.



\section{Methodology}

\subsection{Codebase Integration: Unifying Two Independent HLS Pipelines}

A prerequisite for a controlled comparison between static and dynamic
scheduling is that both pipelines share a common toolchain foundation,
ensuring that observed differences in generated hardware quality are
attributable to the scheduling paradigm rather than to backend divergence.
Achieving this required reconciling two independent research projects—
Dynamatic~v2.0 (pre-release v0.3)~\cite{10.1145/3373087.3375391} and an
incomplete HIR-based static scheduling pipeline~\cite{10.1145/3623278.3624767}—
built on incompatible snapshots of the MLIR/CIRCT
ecosystem~\cite{circt,lattner2021mlir}.

Each project maintained its own fork of both the CIRCT compiler
infrastructure and the Polygeist C-to-MLIR frontend, with bespoke dialect
extensions, pass modifications, and pinned dependency versions that could
not simultaneously coexist. Resolving these incompatibilities required a
systematic four-way codebase integration: the Dynamatic forks of CIRCT and
Polygeist were adopted as the authoritative baselines, and all HIR-specific
dialect definitions, scheduling passes, and lowering infrastructure were
ported forward onto those baselines. This involved updating CMake
configurations and LLVM API call sites to match Dynamatic's pinned versions,
ensuring non-conflicting dialect registration between the \texttt{HIR} and
\texttt{Handshake} dialects, and reconciling differences in the affine IR
shapes emitted by the two Polygeist forks.

Following this integration, both pipelines build and execute within a single
CMake environment, providing the unified compiler on which I conduct all
subsequent experiments in this study.

\subsection{HLS Pragma Annotations for the HIR Pipeline}

The HIR frontend extends the Polygeist C-to-MLIR ingestion pass with a
custom \texttt{\#pragma HLS} handler that recognises six directives,
divided into \emph{region pragmas} (attached to loop or function bodies)
and \emph{variable pragmas} (keyed to a named identifier via a
\texttt{variable=} or \texttt{port=} clause).

\paragraph{Region pragmas.}
\texttt{\#pragma HLS PIPELINE II=\textit{n}} attaches the attribute
\texttt{hls.PIPELINE\_II} to the enclosing \texttt{affine.for} operation;
during the \texttt{AffineToHIRPrep} pass this is renamed to the HIR
attribute \texttt{II} and consumed by the ILP-based scheduler to
constrain the \texttt{hir.next\_iter} offset, enforcing the
user-specified Initiation Interval directly in the IR.
\texttt{\#pragma HLS UNROLL} causes the loop induction variable to be
treated as an \texttt{index} (compile-time constant) type, a prerequisite
for memory banking on any array dimension iterated by the unrolled loop.

\paragraph{Variable pragmas.}
\texttt{\#pragma HLS INTERFACE port=\textit{p} storage\_type=\{ram\_1p
$|$ ram\_2p\} rd\_latency=\textit{r} wr\_latency=\textit{w}} specifies
the memory-interface contract for an external buffer argument; the
\texttt{AffineToHIRPrep} pass translates it into a
\texttt{hir.memref.ports} array attribute that drives the synthesis of
address, data, and enable buses in the generated SystemVerilog.
\texttt{\#pragma HLS BIND\_STORAGE variable=\textit{v}
impl=\{bram$|$lutram\} rd\_latency=\textit{r} wr\_latency=\textit{w}}
applies the same latency semantics to a locally allocated
\texttt{memref.alloca}, additionally specifying the target FPGA memory
primitive via the \texttt{mem\_kind} attribute consumed by the buffer
allocation pass.
\texttt{\#pragma HLS ARRAY\_PARTITION variable=\textit{v} factor=\textit{f}}
requests that a local array be partitioned into $f$ independent banks,
enabling the compiler's banking-analysis pass to mark the corresponding
\texttt{memref} dimension with the \texttt{bank} keyword and thereby
support $f$ parallel accesses per cycle.
Finally, \texttt{\#pragma HLS EXTERN\_FUNC variable=\textit{f}
latency=\textit{l}} annotates an externally provided Verilog IP (e.g.\
a floating-point unit) with its pipeline depth; the value is lowered to
a \texttt{hir.delay} attribute via the \texttt{result\_delays} annotation,
allowing the ILP scheduler to insert the correct number of pipeline stages
between the call site and any consuming operation.
% ----------------------------------------------------------
\subsection{The \texttt{AffineToHIRPrep} Pass}
\label{sec:affine-to-hir-prep}
% ----------------------------------------------------------

A third contribution of this work is the \texttt{AffineToHIRPrep}
MLIR pass. This pass runs immediately before the main \texttt{AffineToHIR}
dialect conversion and serves as a normalisation and annotation
layer between the \texttt{hls.*} attribute vocabulary emitted by the
pragma handler and the HIR-native attribute vocabulary expected by
the downstream conversion.

The pass operates in three sequential stages.
First, it normalises all \emph{external} function declarations by
renaming \texttt{hls.INTERFACE\_LATENCY} to \texttt{hir.delay} on
arguments and results, synthesising default argument names where
absent, and hoisting the declarations to the top of the module so
that every call site can resolve its callee during conversion.
Second, it processes each non-external function by translating the
\texttt{hls.INTERFACE\_*} and \texttt{hls.BIND\_STORAGE\_*}
attributes placed by the pragma handler into HIR's
\texttt{hir.memref.ports} port-descriptor arrays and
\texttt{mem\_kind} implementation attributes, renaming
\texttt{hls.PIPELINE\_II} to the \texttt{II} attribute directly
consumed by the ILP scheduler, and annotating every
\texttt{affine.load} and \texttt{affine.store} with
\texttt{result\_delays} and \texttt{hir.memref\_port} indices
derived from the port descriptors.
It additionally promotes rank-0 \texttt{memref} values---arising from
scalar C variables that Polygeist lowers to zero-dimensional
buffers---to \texttt{memref<1xT>} with register-style port
annotations, as HIR has no representation for zero-dimensional
memories.
Finally, all intermediate \texttt{hls.*} attributes are stripped,
leaving only well-formed HIR attributes for the subsequent
conversion pass.
Together, these transformations eliminate the need for manual
post-hoc IR editing that the original HIR compiler flow
required~\cite{10.1145/3623278.3624767}, and ensure that the
\texttt{AffineToHIR} conversion receives a structurally valid
and fully annotated module.
